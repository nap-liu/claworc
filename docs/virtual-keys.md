# LLM Gateway Virtual Keys

## Overview

When an OpenClaw instance needs to call a real LLM provider (Anthropic, OpenAI, Ollama, etc.),
it does not receive the raw provider API key. Instead, the control plane issues each instance a
**virtual key** — a short-lived, per-instance, per-provider credential with the `claworc-vk-*` prefix.

The instance sends its virtual key to an internal LLM gateway running on the control plane. The
gateway validates the token, resolves the real provider API key, and proxies the request. Real
API keys never leave the control plane.

Benefits:
- Provider credentials stay outside containers (no risk of exfiltration via container breakout or logs).
- Per-instance keys can be revoked individually without rotating the real provider key.
- All proxied requests are logged for usage tracking and auditing.
- The gateway endpoint (`127.0.0.1:<port>`) is only reachable from inside the container via an
  SSH tunnel, never exposed publicly.


## Architecture

```
OpenClaw container
 └─ openclaw (uses claworc-vk-* token via Bearer / x-api-key / x-goog-api-key depending on SDK)
     │
     │  HTTP POST http://127.0.0.1:<port>/...
     │
     ▼
SSH agent-listener tunnel (port <port> on container)
     │
     │  Forwarded over the control plane's SSH connection to the instance
     │
     ▼
Control plane — LLM gateway (127.0.0.1:<port>)
 1. Validate gateway token →  look up LLMGatewayKey in SQLite
 2. Resolve real API key   →  instance override or global setting
 3. Proxy request          →  forward to provider, setting correct auth header for api_type
 4. Log request            →  write LLMRequestLog row
     │
     ▼
Real LLM provider (Anthropic, OpenAI, Google, …)
```

The gateway port defaults to `40001` and is configurable via the `CLAWORC_LLM_GATEWAY_PORT`
environment variable.

### SSH agent-listener tunnel

The gateway listens on `127.0.0.1:<port>` on the control plane — it is not publicly exposed.
Containers reach it via a **`TunnelTypeAgentListener`** SSH tunnel: the control plane uses the
existing per-instance SSH connection to tell the SSH server (agent) to listen on the configured
port and forward incoming connections back to `127.0.0.1:<port>` on the control plane.

This tunnel is created automatically by `TunnelManager.CreateAgentListenerTunnel` when tunnels
are reconciled for an instance that has at least one enabled provider.

Source: `control-plane/internal/sshproxy/tunnel.go` — `CreateAgentListenerTunnel`.

### Virtual key format

```
claworc-vk-<48 hex chars>
```

Generated by `generateGatewayKey()` using `crypto/rand`. Each key is unique, stored in the
`llm_gateway_keys` table, and maps exactly one instance to one provider.

Source: `control-plane/internal/llmgateway/keys.go`.


## How Key Resolution Works

When the gateway receives a request, `authAndResolve` looks up the `claworc-vk-*` token in
`llm_gateway_keys`, loads the associated `LLMProvider`, then calls `resolveRealAPIKey`:

```go
// gateway.go: resolveRealAPIKey
keyName := strings.ToUpper(strings.ReplaceAll(providerKey, "-", "_")) + "_API_KEY"
// e.g. provider key "anthropic" → "ANTHROPIC_API_KEY"
//      provider key "my-ollama" → "MY_OLLAMA_API_KEY"

// Global setting (settings table, "api_key:" prefix)
if found in settings WHERE key = "api_key:" + keyName {
    return decrypt(value)
}

return ""  // no key found — request will be rejected by provider
```

The key is stored Fernet-encrypted at rest in the global settings table.


## How OpenClaw Receives the Config

All proxied models are configured as "custom" in OpenClaw as described on
https://docs.openclaw.ai/concepts/model-providers#local-proxies-lm-studio-vllm-litellm-etc, even well known, 
like Anthropic and OpenAI.

When a provider is enabled for an instance, `ConfigureInstance` pushes two things into the
container via SSH exec (running as the `claworc` user):

### 1. Model config

```
openclaw config set agents.defaults.model '{"primary":"<model>","fallbacks":[...]}'
```

Sets the primary model and fallback list for default agents.

### 2. Gateway providers → `models.providers`

```
openclaw config set models.providers '...' --json
```

Each enabled provider produces an entry like:

```json
{
  "anthropic": {
    "baseUrl": "http://127.0.0.1:40001",
    "api":     "anthropic-messages",
    "apiKey":  "claworc-vk-<token>",
    "models": [
      {
        "id": "claude-sonnet-4-6",
        "name": "Claude Sonnet 4.6",
        "contextWindow": 200000,
        "maxTokens": 64000,
        "cost": {"input": 3.0, "output": 15.0, "cacheRead": 0.3, "cacheWrite": 3.75}
      }
    ]
  }
}
```

- `api` matches the provider's `api_type` field (e.g. `anthropic-messages`, `openai-completions`)
- `models` contains only the models enabled for this specific instance, with full metadata
- `apiKey` is the virtual key (`claworc-vk-*`) — the real key never leaves the control plane

**Note on model discovery:** Because all Claworc-managed providers point to the local LLM gateway
(`http://127.0.0.1:<port>`) rather than a first-party upstream URL, OpenClaw treats every configured
provider as a **custom provider**. OpenClaw only discovers models for custom providers from the
`models` array in the provider config — it does not perform catalog discovery for them. Therefore
the `models` array is always populated with the full definition (id, name, cost, context window,
reasoning, etc.) for each model enabled on the instance.

After writing the config, `openclaw gateway stop` is called so OpenClaw picks up the new settings.

Source: `control-plane/internal/handlers/instances.go` — `ConfigureInstance`.


## Supported API Types

The `api_type` field on an `LLMProvider` controls how the gateway forwards auth and how OpenClaw
calls the provider.

| `api_type` | Client lib (OpenClaw) | Auth sent to gateway | Base URL format | Example base URL |
|---|---|---|---|---|
| `openai-completions` | openai SDK `/chat/completions` | `Authorization: Bearer` | ends with `/v1` | `https://api.openai.com/v1` |
| `openai-responses` | openai SDK `/responses` | `Authorization: Bearer` | ends with `/v1` | `https://api.openai.com/v1` |
| `openai-codex-responses` | openai SDK `/responses` | `Authorization: Bearer` | ends with `/v1` | — |
| `anthropic-messages` | @anthropic-ai/sdk `/v1/messages` | `x-api-key` | root URL (no `/v1`) | `https://api.anthropic.com` |
| `google-generative-ai` | @google/genai SDK | `x-goog-api-key` | ends with `/v1beta` | `https://generativelanguage.googleapis.com/v1beta` |
| `github-copilot` | @anthropic-ai/sdk | `Authorization: Bearer` | dynamic | — |
| `bedrock-converse-stream` | AWS SDK | `Authorization: Bearer` | AWS region URL | `https://bedrock-runtime.{region}.amazonaws.com` |
| `ollama` | direct fetch `/api/chat` | `Authorization: Bearer` (empty ok) | root URL | `http://localhost:11434` |

### Auth header notes

- The gateway accepts the `claworc-vk-*` virtual key in any of these locations (tried in order):
  `Authorization: Bearer`, `x-api-key`, `x-goog-api-key`, `?key=` query param.
- The gateway always strips all incoming auth headers before forwarding, then sets the correct
  outgoing auth header based on `api_type`.

### Base URL notes

- **`anthropic-messages`**: set `base_url` to the root (e.g. `https://api.anthropic.com`) — the
  Anthropic SDK appends `/v1/messages` itself. Do **not** include `/v1` in the base URL.
- **`google-generative-ai`**: set `base_url` to include `/v1beta` (e.g.
  `https://generativelanguage.googleapis.com/v1beta`) — the Google SDK omits the version when a
  custom `baseUrl` is provided.
- **`openai-completions` / `openai-responses`**: the gateway automatically de-duplicates `/v1`
  if `base_url` already ends with `/v1`.


## Testing with Virtual Keys

This walkthrough shows how to exercise the gateway without a real LLM provider.

### Prerequisites
- Control plane running locally (port 8080).
- At least one instance running and SSH-connected.

### Step 1: Create a test provider pointing at httpbin

```bash
curl -s -X POST http://localhost:8080/api/v1/providers \
  -H 'Content-Type: application/json' \
  -d '{"key":"test","name":"Test","base_url":"https://httpbin.org/anything"}'
```

Note the returned `id` (e.g. `1`).

### Step 2: Store a dummy API key

```bash
curl -s -X PUT http://localhost:8080/api/v1/settings \
  -H 'Content-Type: application/json' \
  -d '{"api_key:TEST_API_KEY":"dummy-key"}'
```

### Step 3: Enable the provider for an instance

```bash
curl -s -X PUT http://localhost:8080/api/v1/instances/1 \
  -H 'Content-Type: application/json' \
  -d '{"enabled_providers":[1]}'
```

### Step 4: Find the virtual key in SQLite

```bash
sqlite3 /app/data/claworc.db \
  "SELECT gateway_key FROM llm_gateway_keys WHERE instance_id=1 AND provider_id=1;"
# → claworc-vk-<token>
```

### Step 5: Call the gateway directly (from the control plane host)

```bash
curl -s http://127.0.0.1:40001/v1/chat/completions \
  -H "Authorization: Bearer claworc-vk-518e7917958e3b464d95eee0e91a41d9804f05c9b4c2f377" \
  -H 'Content-Type: application/json' \
  -d '{"model":"test","messages":[{"role":"user","content":"hi"}]}'
```

The gateway proxies to `https://httpbin.org/anything/v1/chat/completions`, which echoes back
the request — you should see your payload and the `Authorization: Bearer dummy-key` header that
the gateway substituted.

### Step 6: Verify the log entry

```bash
curl -s 'http://localhost:8080/api/v1/usage-logs?instance_id=1' | jq '.[0]'
```


## Usage Logs

Every proxied request is logged to the `llm_request_logs` table in a dedicated **`{DataPath}/llm-logs.db`** SQLite file (separate from `claworc.db` to keep the main database lean and allow independent log rotation).

### Endpoint

```
GET /api/v1/usage-logs
```

**Query parameters:**

| Parameter | Type | Description |
|-----------|------|-------------|
| `instance_id` | int | Filter by instance |
| `provider_id` | int | Filter by provider |
| `limit` | int | Max results (default 100, max 1000) |
| `offset` | int | Pagination offset |

Results are ordered newest-first.

**Response fields:**

| Field | Description |
|-------|-------------|
| `id` | Log entry ID |
| `instance_id` | Instance that made the request |
| `provider_id` | Provider the request was routed to |
| `provider_key` | Provider slug (e.g. `anthropic`) |
| `model_id` | Model name from the request body |
| `input_tokens` | Prompt token count (non-streaming only) |
| `output_tokens` | Completion token count (non-streaming only) |
| `cached_input_tokens` | Cached prompt tokens (non-streaming only; 0 if provider doesn't report cache hits) |
| `cost_usd` | Estimated cost in USD based on model cost config (0 if model cost not configured) |
| `status_code` | HTTP status returned by the upstream provider |
| `latency_ms` | End-to-end latency in milliseconds |
| `error_message` | First 500 bytes of error body (on 4xx/5xx) |
| `requested_at` | UTC timestamp |

Streaming responses (`text/event-stream`) are logged with `input_tokens=0, output_tokens=0, cached_input_tokens=0, cost_usd=0`
because the response body is streamed directly without buffering.

**Cost formula:**

```
cost_usd = (
    (input_tokens - cached_input_tokens) × input_rate +
    cached_input_tokens                  × cache_read_rate +
    output_tokens                        × output_rate
) / 1_000_000
```

Rates come from the `cost` field of the matching model in the provider's `Models` config. If no cost config is found for the model, `cost_usd` is `0`.
